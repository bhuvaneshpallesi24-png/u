02 DOCKER IMAGE LAYERS — docker history
🧠
Core Rule: Every line in docker history = one layer. Every Dockerfile instruction creates a layer. Image = stack of layers, bottom to top. Layers with size 0B are metadata only (CMD, ENV, LABEL, ARG).
🐳 Alpine — 2 Layers (Simplest)
docker history alpine
0B
CMD ["/bin/sh"] — runtime instruction (metadata only)
8.44MB ←
ADD alpine-minirootfs.tar.gz / — the actual OS filesystem
✅
Why only 2 layers? Alpine's philosophy is minimalism. One layer for the OS, one for the default command. Nothing else. This is why it's 8MB and pulls so fast.
🐳 Ubuntu — ~6 Layers
docker history ubuntu
0B
CMD ["/bin/bash"]
78.1MB ←
ADD file: rootfs.tar.xz — full Ubuntu OS
0B
LABEL org.opencontainers...
0B
ARG RELEASE / ARG LAUNCHPAD... — build args (metadata)
💡
Key insight: Only 1 heavy layer (78MB ADD). The rest are 0B metadata. Total layers ~6 but only one has real size. LABELs and ARGs add zero bytes.

Why node:latest = 1.13GB? Each RUN apt-get creates a separate layer. Multiple installs = multiple large layers stacked. Every RUN command is a new layer — even if they install the same packages. This is why you should chain RUN commands with && in production Dockerfiles.

03 DOCKER LAYER CACHING — MOST IMPORTANT
🔥 The One Rule to Rule Them All
If a layer changes → ALL layers below it are rebuilt
Docker builds top → bottom. Cache hit on every layer until the first change. Once a layer is invalidated, every layer after it must be rebuilt — no exceptions.
✅ Good Dockerfile — Cache-Optimized
# 1. Base image — rarely changes FROM node:18-alpine WORKDIR /app # 2. Copy only deps file FIRST COPY package.json . # 3. Install — heavy, but cached RUN npm install # 4. Copy code LAST (changes often) COPY . . CMD ["node", "app.js"]
Case 1 — Only app.js changes
1
FROM node:18-alpine ✅ CACHED
2
COPY package.json . ✅ CACHED — file unchanged
3
RUN npm install ✅ CACHED — no reinstall!
4
COPY . . 🔄 REBUILD — code changed
⚡ Build time: ~5 seconds (no npm install)
Case 2 — package.json changes (new dependency)
1
FROM node:18-alpine ✅ CACHED
2
COPY package.json . 🔄 CHANGED — new dep added
3
RUN npm install 🔄 MUST RERUN
4
COPY . . 🔄 REBUILD
🐢 Build time: 2–5 min (npm install runs again — correct behaviour)
❌ Bad Dockerfile — Cache-Broken
# WRONG — code copied before install FROM node:18-alpine WORKDIR /app # Code first (changes every time!) COPY . . # Install AFTER code — broken cache RUN npm install CMD ["node", "app.js"]
What happens when you change app.js
1
FROM node:18-alpine ✅ CACHED
2
COPY . . 🔄 ANY FILE CHANGE = MISS
3
RUN npm install 🔄 ALWAYS REINSTALLS ❌
🐢 Build time: 5–10 min EVERY TIME (even for 1 line change)
The Lunchbox Analogy (your understanding)
📦 Packing lunch in steps: rice → curry → salad → dessert
If you change only dessert → rice, curry, salad unchanged
✅ Same in Docker: change code (last step) → only last layer rebuilds
❌ Put changing thing first → everything after it rebuilds
🎯
Your polished insight: "Stable, rarely-changing layers (base image, dependencies) go at the top. Frequently-changing layers (app code) go at the bottom. This maximizes cache hits and keeps CI/CD builds fast."
Layer TypeChanges How OftenPositionCI/CD Impact
Base image (FROM)Almost neverTop ✅Always cached
package.json / requirements.txtOccasionally (new deps)Near top ✅Cache until dep changes
RUN npm install / pip installWith dep fileAfter dep file ✅Cached most builds
COPY . . (app code)Every commitBottom ✅Only this layer rebuilds
💬 "Why COPY package.json before COPY . .?" — To leverage layer caching. Dependencies change rarely; code changes every commit. Putting stable layers first avoids reinstalling npm packages on every code change, reducing build time from minutes to seconds.
04 MULTI-STAGE BUILDS — PRODUCTION LEVEL
❌ Problem — Single Stage
FROM node:18 WORKDIR /app COPY . . RUN npm install RUN npm run build CMD ["node", "app.js"]
# Final image contains: ❌ node_modules (dev deps) ❌ Source code ❌ Build tools (compilers) ❌ npm cache Result: 1GB+ image 😱
✅ Solution — Multi-Stage
# Stage 1: Build FROM node:18 AS builder WORKDIR /app COPY package.json . RUN npm install COPY . . RUN npm run build
# Stage 2: Run (clean slate) FROM node:18-alpine WORKDIR /app COPY --from=builder /app/dist ./dist CMD ["node", "dist/app.js"]
# Final image contains: ✅ Only dist/ files Result: ~100MB ✅

🏗️ House-Building Analogy (remember this forever)

Cranes, cement mixers, scaffolding, workers, raw materials — everything needed to BUILD the house.

Build tools are ONLY in Stage 1 — they never appear in the final image.
Real Company Patterns
🟢 React App
# Build FROM node:18 AS build RUN npm run build # Serve FROM nginx:alpine COPY --from=build /app/build /usr/share/nginx/html
☕ Java / Spring Boot
# Build FROM maven:3.9 AS builder RUN mvn package # Run FROM eclipse-temurin:17-alpine COPY --from=builder *.jar app.jar
🐍 Python (ML apps)
# Build FROM python:3.10 AS builder RUN pip install --prefix=/install -r requirements.txt # Run FROM python:3.10-slim COPY --from=builder /install /usr/local
QuestionAnswer
Where are build tools in final image?NOT there. Second FROM starts a completely fresh image.
What does --from=builder do?Copies files from a previous stage into the current stage.
When is multi-stage most useful?React builds, Java JAR builds, Go binaries — when build output ≠ source code.
Security benefit?Source code, secrets, build credentials never reach the final image.
Size reduction for Java?800MB+ (maven) → ~200MB (JRE only). ~75% reduction.

06 NODE.JS DOCKERFILE — NORMAL & MULTI-STAGE
Normal Dockerfile
# Base image (alpine = lightweight) FROM node:18-alpine # Set working directory WORKDIR /app # Copy dep file FIRST (caching) COPY package.json . # Install RUN npm install # Copy code COPY . . # Expose (good practice) EXPOSE 3000 # Run CMD ["node", "app.js"]
Build & Run
# ⚠️ Common mistake — missing dot! docker build -t node-app . # The . = current directory (build context) # Without it: ERROR requires 1 argument
docker run -d -p 3000:3000 node-app
Multi-Stage Dockerfile (Production)
# ─── Stage 1: Build ─── FROM node:18 AS builder WORKDIR /app COPY package.json . RUN npm install COPY . . # If your app has build step: # RUN npm run build
# ─── Stage 2: Run ─── FROM node:18-alpine WORKDIR /app # Copy ONLY what you need to run COPY --from=builder /app/dist ./dist EXPOSE 3000 CMD ["node", "dist/app.js"]
💡
When does multi-stage help for Node? Most useful when you have a build step (TypeScript → JS, React → static). For a simple Node app with no build step, the size reduction is minimal — but it still separates concerns cleanly.
⚠️
Debug you hit: docker build -t node-app — missing the . at the end. The dot tells Docker to use the current directory as the build context. Without it, Docker doesn't know where your Dockerfile and code are.

08 JAVA DOCKERFILE — REAL DEBUGGING
Normal Dockerfile (Learning / Local)
# Maven image = Java + build tools FROM maven:3.9.9-eclipse-temurin-17 WORKDIR /app COPY . . RUN mvn clean package -DskipTests EXPOSE 8080 CMD ["java", "-jar", "target/docker-demo-1.0.jar"]
# ⚠️ Problem: ~800MB+ (maven included)
Multi-Stage Dockerfile (Production ✅)
# ─── Stage 1: Build ─── FROM maven:3.9.9-eclipse-temurin-17 AS builder WORKDIR /app # Copy pom FIRST (dep caching) COPY pom.xml . RUN mvn dependency:go-offline COPY . . RUN mvn clean package -DskipTests
# ─── Stage 2: Run ─── FROM eclipse-temurin:17-jdk-alpine WORKDIR /app COPY --from=builder /app/target/*.jar app.jar EXPOSE 8080 CMD ["java", "-jar", "app.jar"]
Bug 1 — no main manifest attribute
docker logs java-app no main manifest attribute, in target/docker-demo-1.0.jar
Root cause: Spring Boot plugin missing in pom.xml → JAR not executable → no Main-Class in META-INF/MANIFEST.MF
Fix — Add to pom.xml
<build> <plugins> <plugin> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-maven-plugin</artifactId> </plugin> </plugins> </build>
Bug 2 — Unable to find main class
Unable to find main class → [Help 1] spring-boot-maven-plugin:repackage failed
Root cause: @EnableAutoConfiguration alone is not enough. Spring Boot plugin needs @SpringBootApplication to auto-detect the entry point.
Fix — Update Java code
// Wrong: @EnableAutoConfiguration public class Application { ... }
// Correct: @RestController @SpringBootApplication public class Application { ... }
🧠
SRE Debug Mindset: Container exits → check logs → error is from app (not infra) → fix app code → rebuild. Not every container failure is an infrastructure issue. Know when it's code vs config vs infra.

📄 What is pom.xml? — Deep Explanation
pom.xml = Project Object Model
Think of it as the Java equivalent of package.json for Node or requirements.txt for Python — but more powerful. It defines everything about your Java project.

Key sections of pom.xml

<dependencies> ← like npm deps <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> </dependencies>
<build><plugins> ← build tools spring-boot-maven-plugin ← makes JAR executable </plugins></build>
📦 What is a JAR file?
JAR = Java Archive
A JAR is essentially a ZIP file that contains all your compiled Java classes, dependencies, and a metadata file called META-INF/MANIFEST.MF. When you run java -jar app.jar, the JVM reads the manifest to find the entry point class.
🍱 Cooking analogy:
Node/Python: "Here are the raw ingredients (source files), cook as you go."
Java: "Here's the pre-cooked meal in a box (JAR), just heat and serve."
Inside a JAR file
app.jar (ZIP file containing:) ├── META-INF/ │ └── MANIFEST.MF ← entry point │ Main-Class: com.example.Application ├── com/example/ │ └── Application.class ← compiled code ├── org/springframework/ │ └── ... ← Spring deps └── application.properties ← config
# Run it: java -jar app.jar # JVM reads MANIFEST → finds Main-Class → executes
QuestionAnswer
Why Node/Python don't need compilation?They're interpreted — the runtime reads and executes source code directly, no intermediate binary needed.
What happens inside a JAR?Compiled .class files + dependencies + MANIFEST.MF (entry point) — everything bundled as a ZIP.
Why Java needs Maven?Java is a compiled language. Maven compiles .java → .class, resolves dependencies, and packages everything into a JAR.
Which step happens inside docker build vs run?docker build = RUN instructions (install, compile). docker run = CMD (start application).
11 docker image inspect + docker rmi
🧠 What is docker image inspect?
Returns the full JSON metadata of an image — exposed ports, environment variables, default command, entrypoint, volumes, architecture, and more. Essential for understanding what an image will do before running it.
3 Fields You Must Know
🔌 1. ExposedPorts
docker image inspect nginx | grep -i ExposedPorts -A 5
# Output: "ExposedPorts": { "80/tcp": {} }
Container listens on port 80 internally.
EXPOSE ≠ actually open port. You still need -p 80:80 at runtime.
🌍 2. Env (Environment Variables)
docker image inspect nginx | grep -i Env -A 10
# nginx output: "Env": [ "PATH=/usr/local/sbin:...", "NGINX_VERSION=1.25.x" ]
Default environment variables baked into the image. Used for config, runtime behavior, binary paths.
▶️ 3. Cmd (MOST IMPORTANT)
docker image inspect nginx | grep -i Cmd -A 5
# Output: "Cmd": [ "nginx", "-g", "daemon off;" ]
This is why docker run nginx automatically starts nginx in foreground — the CMD is embedded in the image.
Debug — You Inspected the Wrong Image
You ran docker image inspect and got JAVA_HOME, MAVEN_HOME — that's the Java image, not nginx. Always confirm which image you're inspecting.
# ❌ You got (Java image): "JAVA_HOME=/opt/java/openjdk" "MAVEN_HOME=/usr/share/maven"
# ✅ nginx correct output: "NGINX_VERSION=1.25.x"
# Clean way — grep specific fields at once: docker image inspect nginx | grep -E "ExposedPorts|Cmd|Env" -A 5
🚨 Real SRE Use Cases for inspect
⚠️ Container not starting → check Cmd field for wrong command
⚠️ App needs config → check Env to see what vars the image expects
⚠️ Port confusion → check ExposedPorts to know what port the app uses internally
✅ Validate unknown image → inspect before running in production
💬 "docker image inspect provides metadata: exposed ports, env vars, CMD, entrypoint, architecture — critical for debugging containers before they run."

⚡ docker exec — Enter Running Containers (VERY IMPORTANT)
Full Lab Sequence
# Container must be running first docker run -d --name nginx-d -p 8081:80 nginx
# Enter the running container docker exec -it nginx-d bash
# Inside container — modify nginx page root@abc:/# cd /usr/share/nginx/html root@abc:/# echo "Hello from SRE 🚀" > index.html root@abc:/# exit
# See your change in browser: # http://<EC2-IP>:8081
docker run -it vs docker exec — KEY DIFFERENCE
docker run -it
Creates a new container and enters it. Container stops on exit.
docker exec -it
Enters an already running container. Container keeps running on exit.
Useful exec Commands
# Run single command without entering shell docker exec nginx-d cat /etc/nginx/nginx.conf
# Check processes inside container docker exec nginx-d ps aux
# Check env inside running container docker exec nginx-d env
🚨
SRE Production Use: In production, you never recreate a container to debug. You docker exec into it, investigate logs/config/processes live, fix what you can, then exit. Container keeps serving traffic throughout.
📋 docker cp — Copy Files Between Host ↔ Container
Host → Container
# Create file on host echo "Copied from host" > test.txt
# Copy to container docker cp test.txt nginx-d:/usr/share/nginx/html/
# Verify inside container docker exec nginx-d ls /usr/share/nginx/html index.html test.txt
Container → Host
docker cp nginx-d:/usr/share/nginx/html/index.html . ls index.html ← pulled from container
Syntax Pattern
docker cp source destination
Host→Container: localfile name:/path
Container→Host: name:/path ./localpath
🚨 Real SRE Uses for docker cp
  • Emergency config update without rebuild
  • Extract crash dump / log file from dead container
  • Inject a hotfix script into running container
  • Pull generated reports or artifacts from container
🧹 --rm Flag — Auto-Cleanup
Without --rm (default)
docker run ubuntu echo "temporary"
docker ps -a CONTAINER ID STATUS abc123 Exited (0) ← still there! def456 Exited (0) ← accumulates ghi789 Exited (0) ← disk waste
With --rm (clean)
docker run --rm ubuntu echo "temporary" temporary
docker ps -a # Nothing — container auto-deleted ✅
✅ CI/CD Pipelines
Run tests in container → container auto-removes → no cleanup script needed
✅ One-Off Tasks
Database migrations, script runners, batch jobs — run and vanish cleanly
✅ Disk Health
Prevents dead containers piling up and wasting disk on /var/lib/docker
❌ Don't use for
Long-running services — container removed means you lose logs forever
🚨 SRE Challenge 1 — Crash a Container (kill PID 1)
The Experiment
# Run nginx docker run -d --name crash-nginx nginx
# Enter container docker exec -it crash-nginx bash
# Inside: kill PID 1 (nginx master process) root@abc:/# kill 1 root@abc:/# exit
# Check status docker ps -a crash-nginx Exited (137) ← killed by signal
What Just Happened?
1
kill 1 sends SIGTERM to PID 1
nginx master process (PID 1 inside container) receives signal
2
PID 1 exits
The main process of the container terminates

🏆 Interview Q&A — Container Modes & Operations
Q. What happened when you ran kill 1 inside the container?
Sending kill 1 inside the container kills the nginx master process (PID 1). Since a container's lifecycle is tied to PID 1, the container immediately stopped with exit code 137 (killed by signal).
Q. Why is --rm useful in CI/CD?
CI/CD jobs are temporary — build, test, done. Without --rm, hundreds of exited containers pile up on the build host, consuming disk space. --rm ensures automatic cleanup after the job finishes, keeping the host clean without extra maintenance scripts.
Q. Difference between docker exec and docker run -it?
docker run -it creates a new container and drops you into an interactive shell — the container stops when you exit. docker exec -it runs a command inside an already running container without affecting its lifecycle — the container keeps running after you exit.
Q. How do you detect a memory leak in Docker?
I use docker stats to monitor MEM USAGE in real time. If the value keeps climbing without releasing, that's a memory leak. I also check docker inspect for OOMKilled: true and ExitCode: 137 as evidence of past OOM kills.
10 FINAL SRE TAKEAWAYS — DAY 2
📦
Layer = Instruction
Every Dockerfile instruction = 1 layer. 0B layers are metadata (CMD, ENV, LABEL). Only ADD/RUN/COPY add real size. Fewer heavy layers = smaller image.
🚀
Stable Top, Changes Bottom
Rarely changing layers (base, deps) go to the top. Frequently changing (code) at the bottom. One layer change invalidates all below it — structure matters.
🏗️
Multi-Stage = Lean Prod
Stage 1 builds, Stage 2 runs. Build tools never reach the final image. Java: 800MB → 200MB. React: 1GB → 30MB. Always use multi-stage in production.
⚡
RUN vs CMD
RUN executes during build time (install, compile). CMD executes when container starts (run app). Mixing them up is the #1 beginner Docker mistake.
☕
Java Needs Maven
Java is compiled. Source → bytecode → JAR. Maven is the build tool. @SpringBootApplication is mandatory for Spring apps. MANIFEST.MF must have Main-Class.
🔍
Debug = docker logs first
Container exits → check logs. It might be an app bug (wrong annotation) or config issue (missing plugin) — not always infra. Logs tell the truth.
🏆 Interview-Ready Answers — Day 2
Q. "How do you optimize a Dockerfile?"
Use a minimal base image (alpine/slim), leverage layer caching by copying dependency files before source code, chain related RUN commands with &&, use .dockerignore, and use multi-stage builds to exclude build tools from the final image.
Q. "Why is the node:latest image 1.13GB?"
node:latest is based on Debian and includes 12+ layers from multiple apt-get install commands, each creating a separate heavy layer — Debian base (~117MB), system packages (~588MB), Node.js runtime (~198MB), Yarn, and more.
Q. "How do you write a Dockerfile for any application?"
I ask three questions: what is the runtime (node/python/java), what are the dependencies (package.json/requirements.txt/pom.xml), and what is the start command. Then: FROM runtime → WORKDIR → COPY dep file → RUN install → COPY code → CMD start. For production, wrap in multi-stage.
