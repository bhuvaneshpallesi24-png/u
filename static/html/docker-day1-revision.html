07 STOP vs KILL
docker stop
🛑
SIGTERM
Sends SIGTERM first — asks the process to gracefully shut down. Waits 10 seconds for cleanup. If still alive, sends SIGKILL.
docker stop my-nginx # Exit code: 0 (clean)
✅
Use in production. Allows the app to close DB connections, flush buffers, finish in-flight requests.
docker kill
💀
SIGKILL
Sends SIGKILL immediately — the kernel kills the process instantly. No cleanup possible. App has no say.
docker kill my-nginx # Exit code: 137 (128+9 = SIGKILL)
⚠️
Last resort only. Use when container is frozen and not responding to stop. Risk: data corruption, incomplete writes.
Exit Code Reference — memorize these for SRE interviews
0
Clean exit
docker stop or normal completion
1
App error
Process crashed / non-zero exit
137
SIGKILL / OOM
docker kill or OOMKilled (128 + 9)

Scenario 1 — Intentional Crash
# Run a container that exits with error code 1 docker run --name crash-test ubuntu false docker ps -a crash-test ubuntu Exited (1) 2s ago # Logs show nothing — 'false' command has no output docker logs crash-test
🔍
Exit code 1 = the process failed. The command false always exits with 1. In real life this means your app crashed at startup — check the logs first.
Scenario 2 — Wrong Path Error
# Try to list a directory that doesn't exist docker run --name crash-test2 ubuntu ls /wrong-path docker logs crash-test2 ls: cannot access '/wrong-path': No such file or directory docker inspect crash-test2 --format '{{.State.ExitCode}}' 2
Scenario 3 — Read inspect for full context
docker inspect crash-test2 # Look for these fields in the output: "State": { "Status": "exited", "ExitCode": 2, "Error": "", "OOMKilled": false, "StartedAt": "2024-01-15T10:00:01Z", "FinishedAt": "2024-01-15T10:00:01Z" }
💡
OOMKilled: true means the container was killed by the OS because it exceeded memory limits. This is exit code 137. You'll see this in production when containers grow unbounded.

CORE IDEA
🧠 Remember Forever
🐳
A Container = One Main Process (PID 1)
🟢 Process runs → Container runs
🔴 Process exits → Container stops
⚡ Long-Running Process
Command
docker run nginx
Internal Flow
1. Docker daemon creates container
2. Starts process: nginx
3. nginx becomes PID 1
4. nginx keeps running → container ALIVE ✅
Container
└── nginx (PID 1)
└── keeps running...
STATUS: RUNNING 🟢
⚡ Short-Lived Process
Command
docker run ubuntu ls
Internal Flow
1. Container created
2. Runs: ls → prints directory listing
3. ls finishes → PID 1 exits
4. Container STOPS ❌
Container
└── ls (PID 1)
└── finishes → exits
STATUS: EXITED (0) 🔴
💥 Why This Design?
Docker is NOT a VM — it does NOT run:
  • Full OS with init system
  • Background daemon services
  • Multiple unrelated processes
It runs exactly ONE main process:
  • That process IS PID 1
  • Container lives and dies with it
  • Isolation is at the process level
🏠 Real-world analogy: A container is like a worker hired for one specific job.
A security guard (nginx) — always on duty → container alive.
A delivery person (ls) — delivers the list, job done → worker leaves → container stops.

💣 Critical Edge Case — Background vs Foreground
❌ Wrong Design
CMD ["service", "nginx", "start"]
# What happens: service nginx start ← PID 1 └── starts nginx in background └── service command EXITS └── PID 1 is gone └── Container STOPS ❌
nginx runs in background → PID 1 (service cmd) exits → container stops even though nginx is technically "running"
✅ Correct Design
CMD ["nginx", "-g", "daemon off;"]
# What happens: nginx -g daemon off ← PID 1 └── nginx runs in FOREGROUND └── PID 1 stays alive └── Container STAYS ALIVE ✅
daemon off forces nginx to run in foreground → stays as PID 1 → container lives
🚨
This is a real production gotcha. Many engineers spend hours debugging why their container "starts but immediately exits" — it's almost always a background/foreground issue. The rule: whatever CMD runs must stay in the foreground and be PID 1.
🏆 Interview Gold Answer
Q. "Why did the container stop?"
A Docker container runs a single main process as PID 1. When that process exits — for any reason — the container stops. If the process is long-running (like a web server or API), the container stays alive. If it's short-lived (like a script or ls), the container exits after execution. This is by design — Docker is not a VM, it's a process runner with isolation.
Quick-fire follow-ups:
Q.Why does nginx container run forever?
A.nginx runs in foreground with daemon off — PID 1 never exits.
Q.Why does docker run ubuntu ls exit?
A.ls is PID 1. It prints output and exits → container stops. Exit code 0 (clean).
Q.Give a real example where container SHOULD exit.
A.Database migrations, backup jobs, CI/CD builds — designed to run and exit cleanly (code 0).
09 FIND CONTAINER PID ON HOST
🧠 Core Concept
When you run docker run -d nginx — Docker does NOT create a VM.
It simply: creates a process on the host (nginx) and applies isolation (namespaces + cgroups).
That process has a real PID on your EC2/host — you can find it, monitor it, trace it, and even kill it directly.
Correct Step-by-Step Flow
# Step 1: Start container docker run -d --name pid-test nginx # Step 2a: grep method docker inspect pid-test | grep '"Pid"' "Pid": 45678, # Step 2b: cleaner --format method (preferred) docker inspect -f '{{.State.Pid}}' pid-test 45678 # Step 3: verify on host using that real PID ps -fp 45678 UID PID CMD root 45678 nginx: master process nginx -g daemon off; # Step 4: see container's process view docker top pid-test UID PID CMD root 1 nginx: master process
⚠️
Common mistake: ps -fp 12345 showing nothing means you used a sample number. Always fetch the actual PID dynamically with docker inspect -f '{{.State.Pid}}' — never assume the number.
Same Process — Two Views (Namespace Magic)
🖥️ Host OS View (ps / top)
├── systemd (PID 1)
├── sshd (PID 1234)
├── nginx (PID 45678) ← your container
↕ same process — different namespace view
🐳 Inside Container View (docker top)
└── nginx (PID 1) ← looks like PID 1
PID namespace remaps 45678 → 1 inside the container

🔥 Why Does It Feel Like a Separate System? — Isolation
1. Namespaces (Isolation)
PID namespace — separate process list (sees itself as PID 1)
NET namespace — separate network interface, IP address
MNT namespace — separate filesystem view
UTS namespace — separate hostname
2. Cgroups (Resource Control)
CPU limits — max CPU % this process can use
Memory limits — max RAM allowed (OOMKill if exceeded)
Disk I/O — read/write bandwidth throttling
Network — bandwidth limits per container
Same host → but namespaces create the illusion of a separate environment
and cgroups enforce resource boundaries. That's the entire magic of containers.
🚨 Real SRE Use Cases — Why You Need the Host PID
Case 1 — High CPU Usage
top PID 45678 → 95% CPU 🚨
# Map that PID to a container docker inspect -f '{{.State.Pid}}' pid-test 45678 ← matches! found the culprit
See high CPU in top → match PID → identify which container is misbehaving
Case 2 — Memory Leak
# Monitor memory of specific PID top -p 45678 PID MEM% RES 45678 2% 80MB 45678 8% 400MB ← growing! 45678 18% 900MB ← leak 🚨
Watch RES (resident memory) growing over time = memory leak evidence
Case 3 — Deep Debug with strace
# Trace ALL system calls of the process strace -p 45678 epoll_wait(...) ← waiting read(5, ...) = -1 EAGAIN ← no data connect(...) = -1 ETIMEDOUT ← network stuck!
strace shows every OS-level call the process makes — network, file, memory operations
Case 4 — Last Resort Kill
# Nuclear option — kill the process directly kill -9 45678 # Container will stop immediately docker ps -a pid-test Exited (137) ← killed
Killing host PID = killing the container. Exit code 137 = SIGKILL. Use only when docker kill fails.
🔬 What is strace? — SRE Superpower
strace = system call tracer
Every program (nginx, python, java) talks to the Linux kernel using system calls.
strace intercepts and shows you every single one in real time — file opens, network calls, memory allocation, everything.
Running strace on ls
strace ls # You'll see calls like: open("/etc/ld.so.cache", O_RDONLY) = 3 read(3, ...) ← reading file write(1, "file1 file2") ← printing output close(3) ← closing file exit_group(0) ← clean exit
3 Real Diagnostic Patterns
🚨 App Stuck / Hanging
strace -p <PID> connect(...) = waiting...

⚠️ File Not Found
open("/app/config.yaml") = -1 ENOENT
Missing config file — container can't find it

🏆 Interview-Level Answers
Q. "What is a container internally?"
A container is a process running on the host system, isolated using Linux namespaces (PID, NET, MNT, UTS) and resource-controlled using cgroups. It's not a VM — it shares the host kernel.
Q. "How do you debug a container at the OS level?"
I use docker inspect -f '{{.State.Pid}}' to get the host PID, then use Linux tools: ps to verify the process, top -p to monitor resources, and strace -p for system call tracing when the app is hanging or behaving unexpectedly.
Q. "What is a memory leak and how do you detect it in Docker?"
A memory leak is when a process allocates memory but never frees it, causing continuous growth until the OS kills it. In Docker, I detect it with docker stats (watching MEM% grow) or top -p <PID> on the host. If OOMKilled, docker inspect will show "OOMKilled": true and ExitCode 137.
🧠 Final truth: Docker is not magic.
Container = process + isolation.
That's why it's fast, lightweight, and easy to scale.
10 FINAL SRE TAKEAWAYS
🏗️
Container = Process
A container is not a mini VM. It's a process with restricted visibility via Linux namespaces. When PID 1 exits, the container stops. Period.
🔌
Client-Server Split
Docker CLI talks to dockerd via the Unix socket. The daemon does everything. CLI is just a REST client. This matters for remote Docker and CI/CD.
💾
Images in /var/lib/docker
All images, layers, volumes live here. Disk fills up in production. Use docker system prune to clean. Monitor this path on your servers.
🚦
Stop Before Kill
Always try SIGTERM (stop) first. Give the app a chance to clean up. SIGKILL (kill) is for emergencies only — exit code 137 is your evidence.
🔍
Debug Flow Matters
ps -a → logs → inspect → exec. That's your 3am production incident flow. The faster you run this, the faster you resolve outages.
🌐
Port Mapping = Forwarding
-p 8080:80 means: traffic hitting host port 8080 → forwards to container port 80. The host port must be free. Container port is internal only.
💬 Interview-Ready One-Liners
Q. What is a Docker container?
A. An isolated process running on the host OS, sharing the kernel but with its own namespace for PID, network, filesystem, and IPC.
Q. Difference between docker stop and docker kill?
A. stop sends SIGTERM (graceful, exit 0), kill sends SIGKILL (force, exit 137). Always use stop in production — kill only when container is frozen.
Q. Why is the container lifecycle tied to PID 1?
A. Docker starts the container's main process as PID 1. When it exits, the container stops. There's no init system — the container IS the process.
