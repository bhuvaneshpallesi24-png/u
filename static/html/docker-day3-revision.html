01 WORKDIR — WHAT, WHY & CONVENTIONS
🧠 Core Definition
WORKDIR sets the default working directory inside the container
It's the Linux equivalent of cd /app — but it also creates the directory if it doesn't exist, and all subsequent COPY, RUN, and CMD instructions operate relative to it.
Without WORKDIR — files land in /
FROM python:3.11-slim # No WORKDIR set COPY . . RUN pip install -r requirements.txt
# Files go to: / (root, messy!) /app.py /requirements.txt ← mixed with OS files ❌
With WORKDIR — clean, predictable
FROM python:3.11-slim WORKDIR /app # create + cd into /app COPY . . # → /app/app.py RUN pip install -r requirements.txt CMD ["python", "app.py"]
# Files go to: /app ✅ /app/app.py /app/requirements.txt
Does WORKDIR change per language?
LanguageCommon WORKDIRValid?
Node.js/app✅ Most common
Python/app✅ Standard
Java/app✅ Clean
Old style/usr/src/app✅ Valid, verbose
Custom/myapp✅ Any valid path
Q. If I remove WORKDIR, where will files go?
Files land in / (root directory). Mixed with OS files — messy and can cause conflicts.
Q. Can I use /myapp instead of /app?
Yes — any valid Linux path works. Docker creates it automatically if it doesn't exist. /app is convention, not a rule.
Q. You used /usr/src/app — is that wrong?
Not wrong — older convention. Today's standard is simply /app. Both work identically. Use /app going forward.
💡
Best practice: Use WORKDIR /app for everything. Simple, standard, readable. The path doesn't affect runtime performance. Consistency matters more than path choice.
💬 "WORKDIR sets the working directory inside the container where COPY, RUN, and CMD instructions execute. It creates the path automatically if it doesn't exist."
02 CMD — EXACTLY WHAT IT DOES
🔥 Core Definition
CMD defines what command runs when the container starts
CMD = the main process = PID 1. When CMD exits, the container stops. When CMD runs forever, the container runs forever. This connects directly to Day 1's core rule.
The runtime flow
1. docker run python-app
2. Container starts
3. CMD executes → becomes PID 1
4. Process runs → container ALIVE ✅
5. Process exits → container STOPS ❌
CMD for each language
# Node.js CMD ["node", "app.js"] # Python CMD ["python", "app.py"] # Java CMD ["java", "-jar", "app.jar"] # Flask (production) CMD ["flask", "run", "--host=0.0.0.0"]
CMD formats — exec vs shell
# Exec form ← PREFERRED (no shell) CMD ["python", "app.py"] # → Process is PID 1 directly ✅ # → SIGTERM signals work correctly
# Shell form ← AVOID in production CMD python app.py # → Shell is PID 1, Python is child # → SIGTERM may not reach Python ❌
Why CMD = PID 1 matters
PID 1 receives all OS signals (SIGTERM, SIGKILL). If CMD runs in exec form, your app handles graceful shutdown correctly. Shell form adds an extra process layer.
CMD vs ENTRYPOINT quick rule

ENTRYPOINTContainer startNo (fixed command)
RUNBuild timeN/A
03 EXPOSE + THE 0.0.0.0 PRODUCTION BUG
What EXPOSE does (and doesn't do)
EXPOSE does
Documents which port the container listens on internally. Visible in docker inspect.
EXPOSE does NOT
Actually open the port. You still need -p 5000:5000 at runtime.
# Python Dockerfile — correct setup FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 5000 # document internal port CMD ["python", "app.py"]
# Runtime — the -p actually opens it docker run -d -p 5000:5000 python-app
The Bug You Hit — 127.0.0.1 vs 0.0.0.0
What your logs showed
docker logs my-prod-u * Running on http://127.0.0.1:5000 # ↑ Only accessible INSIDE container # Browser on EC2 IP → ❌ refused
Why 127.0.0.1 fails — Container Networking
Container has its own network namespace
127.0.0.1
= loopback (inside container only)
Browser on EC2 ❌
0.0.0.0
= all interfaces (any network)
Browser on EC2 ✅
The Fix — app.py
# ❌ Wrong — only inside container app.run(debug=True) # → Flask defaults to 127.0.0.1
# ✅ Correct — all interfaces app.run(host="0.0.0.0", port=5000, debug=False) # → Docker can forward external traffic
🚨
Classic production gotcha. App running → but not reachable from browser. Debug flow: docker logs → see 127.0.0.1 → fix bind address → rebuild → done. Also remove debug=True in production — it's a security risk.
MistakeSymptomFix
app.run(debug=True) (no host)Browser cannot reach appAdd host="0.0.0.0"
Missing -p flagConnection refusedUse docker run -p 5000:5000
EC2 Security GroupPort open in container but blockedAdd inbound rule for port in AWS SG
App binding 127.0.0.1Works inside exec, fails from browserBind to 0.0.0.0
04 PORT CONFLICT — port already allocated
Error you hit
docker run -d -p 5000:5000 --name my-prod-u-1 my-u:v2
Error response from daemon: driver failed programming external connectivity on endpoint my-prod-u-1: Bind for 0.0.0.0:5000 failed: port is already allocated.
🧠
Root cause: Only ONE container (or process) can own a host port at a time. Your first container my-prod-u was still running on port 5000. Two containers cannot both listen on the same host port.
3 Solutions
✅ Option 1 — Stop old container (Best)
docker stop my-prod-u docker rm my-prod-u docker run -d -p 5000:5000 my-u:v2
⚡ Option 2 — Use different host port
# Run on host port 5001 instead docker run -d -p 5001:5000 --name my-prod-u-1 my-u:v2 # Access: http://EC2-IP:5001
🔍 Option 3 — Find what's using the port
netstat -tulnp | grep 5000 # or lsof -i :5000 # Then kill the PID
SRE Mental Model — Port Conflict Debugging
1

2

3

4

Port is now free, run new container

05 RUN vs CMD vs ENTRYPOINT — 3 Dockerfiles
RUN
Build time
Executes during docker build. Creates image layers. Install, compile, setup.
CMD
Runtime (default)
Runs when container starts. Can be overridden by args in docker run.
ENTRYPOINT
Runtime (fixed)
Runs when container starts. Cannot be overridden. Always executes.
❌ Dockerfile 1 — Using RUN (WRONG)
FROM python:3.11-slim WORKDIR /app COPY . . RUN python app.py ← WRONG
What actually happened
[7/7] RUN python app.py 17.1s * Running on all addresses (0.0.0.0) * Running on http://127.0.0.1:5000 # Build stalled — server started during BUILD!
Flask started running during docker build. Build gets stuck because the server never exits. Build = wrong. This is the exact bug you experienced live.
Container: exits immediately after build
✅ Dockerfile 2 — Using CMD (CORRECT)
FROM python:3.11-slim WORKDIR /app COPY . . RUN pip install -r requirements.txt CMD ["python", "app.py"] ← CORRECT
Behaviour
Build completes successfully. CMD only executes when docker run is called. Container starts and keeps running as long as Flask is alive.
docker run -d -p 5000:5000 test-cmd # ✅ Flask starts at runtime # Override example: docker run test-cmd echo "hello" hello ← CMD was replaced
Container: stays alive ✅
🔒 Dockerfile 3 — ENTRYPOINT (Fixed)
FROM python:3.11-slim WORKDIR /app COPY . . RUN pip install -r requirements.txt ENTRYPOINT ["python", "app.py"]
Behaviour
Always runs python — cannot be replaced by docker run args. Useful for tools and CLIs where the command must always be the same.
docker run test-entry echo "hello" # ❌ "echo hello" does NOT replace python # It becomes an ARGUMENT to python # → python echo "hello" (likely error)
Container: always runs python, stays alive
InstructionWhen runsPurposeOverridableReal use case
RUNdocker buildInstall, compile, setupN/Apip install, apt-get, mvn package
CMDdocker run (container start)Default app startupYes — docker run img argsnode app.js, python app.py
ENTRYPOINTdocker run (container start)Fixed commandNo — args are appendedCLI tools, scripts with fixed entry
🔥
The bug you hit explained: You wrote RUN python app.py in your Dockerfile. During docker build, Docker executed this — Flask started running, the server never returned control to Docker, and the build either hung for 17 seconds or failed. RUN is for build-time. CMD is for runtime. This is the #1 most common Docker mistake.
💬 "RUN executes during build to create layers. CMD defines the default command at runtime and can be overridden. ENTRYPOINT sets a fixed command that always runs."
06 ADD vs COPY — When to Use Each
FeatureCOPYADD
Copy files/dirs✅✅
Auto-extract .tar.gz❌ No✅ Yes!
Download from URL❌ No✅ Yes
Recommended for?Files & directories (always)Only when tar extraction needed
PredictabilitySimple, explicitExtra magic can surprise you
📋
Best practice rule: Always use COPY. Only switch to ADD when you specifically need tar extraction. ADD's "magic" (auto-extract, URL download) can cause unexpected behavior in production.
💬 "Use COPY for most cases. ADD is only needed when automatically extracting tar archives."
COPY — explicit, predictable
# Copies file exactly as-is COPY app.py /app/app.py COPY requirements.txt . COPY . . # copy all # Result: file in destination ✅ # No surprises
ADD — auto-extract tar
# Create tar file first tar -cvf test.tar app.py # ADD auto-extracts it FROM alpine WORKDIR /app ADD test.tar . # ← extracts automatically # Inside container: /app/app.py ← extracted ✅ # (not /app/test.tar) # ADD from URL (use sparingly) ADD https://example.com/file.tar.gz /app/
07 .dockerignore — Speed & Security
What is .dockerignore?
Like .gitignore but for Docker. It tells Docker what NOT to include in the build context sent to the daemon. Placed in the same directory as your Dockerfile.
Your .dockerignore file
# Dependencies (regenerated inside image) node_modules __pycache__ *.pyc # Version control .git .gitignore # Secrets — CRITICAL .env *.key *.pem # Dev files Dockerfile* README.md *.log
Without
Sends entire folder to daemon
node_modules (100MB+)
.git history
.env secrets
→ Slow build, large context
With
Only source files sent
No giant node_modules
No secrets exposed
→ Fast build, secure
What to IgnoreWhy
node_modulesRegenerated by npm install inside container
.gitHistory not needed in image
.env🚨 SECRETS — never bake into image
*.logDev logs, useless in prod
__pycache__Python cache, regenerated automatically
🔐
Critical security rule: Never put .env files or API keys in Docker images. Use Docker secrets or environment variables at runtime. Images are inspectable — anyone with your image can read baked-in secrets.

09 BUGS YOU HIT — DOCUMENTED FOREVER
Bug 1 — pip install requirements.txt (missing -r)
Error you got
RUN pip install requirements.txt
ERROR: Could not find a version that satisfies the requirement requirements.txt HINT: You are attempting to install a package literally named "requirements.txt"
Root cause & fix
pip treated requirements.txt as a package name to install, not a file to read from. The -r flag means "read FROM file".
CommandMeaning
pip install flaskInstall package named flask ✅
pip install requirements.txtInstall package named "requirements.txt" ❌
pip install -r requirements.txtRead file, install all packages inside ✅
RUN pip install --no-cache-dir -r requirements.txt

Build output you saw

🟢 SRE Challenge 2 — HEALTHCHECK
Add HEALTHCHECK to Dockerfile
FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 5000
# Health check every 30s HEALTHCHECK --interval=30s --timeout=10s \ CMD curl --fail http://localhost:5000 || exit 1
CMD ["python", "app.py"]
Build, run and verify
docker build -t health-app . docker run -d -p 5000:5000 health-app
# Check health status docker inspect <container> # Look for in JSON output: "Health": { "Status": "healthy", "FailingStreak": 0, "Log": [ { "ExitCode": 0, "Output": "HTTP/1.1 200 OK" } ] }

Why HEALTHCHECK matters in production: Kubernetes and Docker Swarm use health status to decide if a container can receive traffic. An unhealthy container gets replaced automatically. Without it, a crashed-but-running container gets traffic.

11 FINAL SRE TAKEAWAYS — DAY 3
📁
WORKDIR = Your App's Home
Use /app always. Language doesn't matter. Docker creates it automatically. All COPY, RUN, CMD execute relative to it. Without it, files land in / (root — messy).
⚡
CMD = Runtime, RUN = Build
RUN installs and compiles during build. CMD starts your app when container runs. Mix them up = app tries to start during docker build. Most common Dockerfile mistake.
🌐
Always Bind 0.0.0.0
127.0.0.1 = loopback = container-only. 0.0.0.0 = all interfaces = externally reachable. Flask defaults to 127.0.0.1. Always override: host="0.0.0.0".
📋
COPY over ADD
Use COPY for everything. ADD only when you need tar auto-extraction. ADD's "magic" (URL fetch, extract) creates unpredictable builds. Explicit beats magic in prod.
🚀
pip install -r not pip install
-r means "read FROM file". Without -r, pip tries to install a package literally named "requirements.txt" — fails instantly. pip install -r requirements.txt is the only correct form.
🔒
.dockerignore = Security
.env files, .git, node_modules must never reach the image. .env in Docker image = your secrets in the registry = major security breach. Always dockerignore.
🏆 Interview-Ready Answers — Day 3
Q. "What is WORKDIR and does it change per language?"
WORKDIR sets the working directory inside the container where subsequent commands execute. I use /app as convention regardless of language — Node, Python, Java all work the same. Docker creates the path automatically if it doesn't exist.
Q. "Difference between RUN, CMD, and ENTRYPOINT?"
RUN executes during docker build to create image layers (installs, compilations). CMD defines the default command at container start and can be overridden with docker run args. ENTRYPOINT is like CMD but cannot be overridden — it always runs, making it ideal for CLI tools.
Q. "Why was the container running but not accessible?"
The Flask app was bound to 127.0.0.1 (loopback), which is only accessible inside the container's network namespace. Changing to 0.0.0.0 makes it listen on all interfaces, allowing Docker's port forwarding to work. Also checked EC2 Security Group to ensure the port was open externally.
