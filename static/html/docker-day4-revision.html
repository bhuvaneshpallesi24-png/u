01 WHY MULTI-STAGE EXISTS
🧠 The Problem Statement
Build tools should NEVER ship to production
Maven, npm, gcc, compilers — these are needed only to BUILD your app, not to RUN it. Including them in the final image wastes 500MB–700MB, increases attack surface, and slows every deployment.
❌ Single-Stage — What Beginners Write
FROM maven:3.9.9-eclipse-temurin-17 WORKDIR /app COPY . . RUN mvn clean package CMD ["java", "-jar", "target/app.jar"]
# Final image contains ALL of this: ❌ Maven (300MB+) ❌ Source code ❌ Build tools ❌ Compiler ❌ .git history Result: 800MB+ 😱
🔒 Security: More packages = more CVEs in image
🐢 Deploy: 800MB pulls every CI/CD cycle
💰 Cost: 10 pods × 800MB = huge registry egress
✅ Multi-Stage — Production Way
# Stage 1 → BUILDS (heavy, temporary) FROM maven:3.9.9-eclipse-temurin-17 AS builder WORKDIR /app COPY pom.xml . RUN mvn dependency:go-offline COPY . . RUN mvn clean package -DskipTests
# Stage 2 → RUNS (light, final) FROM eclipse-temurin:17-jdk-alpine WORKDIR /app COPY --from=builder /app/target/*.jar app.jar EXPOSE 8080 CMD ["java", "-jar", "app.jar"]
# Final image contains ONLY: ✅ JRE runtime (~200MB) ✅ app.jar Nothing else.
🚀 Size: ~200MB (75% reduction from 800MB)
🔒 Security: No Maven, no source code in prod
⚡ Deploy: Fast pull, fast scaling

02 JAVA MULTI-STAGE — PRODUCTION DOCKERFILE
Production Dockerfile — Annotated
# ──────────── STAGE 1: BUILD ──────────── FROM maven:3.9.9-eclipse-temurin-17 AS builder WORKDIR /app # Copy pom FIRST → cache dependency layer COPY pom.xml . RUN mvn dependency:go-offline # ↑ Downloads all deps once, cached on rebuild # Now copy source (changes often) COPY . . # Build JAR, skip unit tests for speed RUN mvn clean package -DskipTests # Output: /app/target/app-x.x.x.jar # ──────────── STAGE 2: RUN ──────────── FROM eclipse-temurin:17-jdk-alpine # ↑ Alpine-based JRE — tiny, secure WORKDIR /app # Wildcard works in COPY, NOT in CMD COPY --from=builder /app/target/*.jar app.jar EXPOSE 8080 # Exec form — SIGTERM reaches JVM CMD ["java", "-jar", "app.jar"]
Build & Run
# Build image docker build -t java-multi . # Run container docker run -d -p 8080:8080 --name java-app java-multi # Compare size to single-stage docker images REPOSITORY TAG SIZE java-single latest 813MB java-multi latest 198MB ← 75% smaller
Why mvn dependency:go-offline? (Cache trick)
Without it: Every rebuild re-downloads ALL Maven dependencies from internet. Build takes 5–10 min every time.
With it: pom.xml is copied FIRST. Dependencies cached. Code changes only rebuild last layers. Same as Node's package.json trick.
Caching strategy — parallel to Node
LanguageDep file firstInstall cmdThen copy code
JavaCOPY pom.xml .mvn dependency:go-offlineCOPY . .
NodeCOPY package.json .npm installCOPY . .
PythonCOPY requirements.txt .pip install -r requirements.txtCOPY . .
🔥
Universal pattern: Always copy the dependency manifest first, install, then copy code. This pattern is identical across all languages — the names just change. Master this once, apply everywhere.
⚡
-DskipTests in production builds? Tests should run in CI pipeline (before Dockerfile), not inside Docker build. Skipping them makes the build 2–5x faster. Your CI should already validate tests passed.

Multi-Stage — node:18 → node:18-alpine

05 REAL JAR BUGS — DOCUMENTED & FIXED
Bug 1 — Wildcard in CMD doesn't expand
Error you got
CMD ["java", "-jar", "target/spring-petclinic-*.jar"]
Error: Unable to access jarfile target/spring-petclinic-*.jar
CMD runs in exec form (no shell). The shell is what normally expands wildcards (*). In exec form, * is treated as a literal character — no file named spring-petclinic-*.jar exists, so Java crashes.
Fix — rename at COPY stage
Wildcard DOES work in Dockerfile COPY — Docker processes it at build time. So rename the JAR during copy and use an exact name in CMD.
# ✅ Wildcard OK in COPY COPY --from=builder /app/target/*.jar app.jar # ✅ Exact name in CMD CMD ["java", "-jar", "app.jar"]
InstructionWildcard works?
COPY *.jar app.jar✅ Yes — Docker processes it
CMD ["java", "-jar", "*.jar"]❌ No — no shell, literal *
RUN cp target/*.jar app.jar✅ Yes — shell available in RUN
Bug 2 — app.jar not found (single-stage)
Error from docker logs
Error: Unable to access jarfile app.jar
Root cause: The single-stage Dockerfile ran mvn clean package which creates target/spring-petclinic-xxx.jar. But CMD was looking for app.jar which was never created.
# What mvn creates: /app/target/spring-petclinic-3.x.x.jar # What CMD expected: /app/app.jar ← DOES NOT EXIST ❌
Fix — copy and rename in RUN
RUN mvn clean package -DskipTests \ && cp target/*.jar app.jar # ↑ Chain RUN to rename after build
CMD ["java", "-jar", "app.jar"] # ↑ Now app.jar exists ✅
🧠
Debug method: When container crashes, run docker run -it <image> sh then ls /app. Verify the JAR exists at the exact path CMD expects. Build success ≠ file is where you think.

# Container crashes immediately? # Run shell instead of CMD docker run -it petclinic sh # Inside container — verify files ls /app app.jar ← ✅ present target/ ← build artifacts (builder stage) # Test the jar directly java -jar /app/app.jar

06 WHAT HAPPENS INSIDE THE BUILD
docker build internals — step by step
Every instruction in the Dockerfile creates a layer inside the builder container. Maven downloads deps, compiles .java → .class files, then packages them into a JAR — all inside a temporary container that is discarded after docker build finishes.
Stage 1 — What Maven does inside Docker
1. FROM maven → pull image with Java + Maven
2. COPY pom.xml → puts manifest in /app
3. mvn dependency:go-offline → downloads all jars from Maven Central
4. COPY . . → copies .java source files
5. mvn clean package → javac compiles → JARs bundled
6. Output: /app/target/spring-petclinic-x.x.x.jar ✅
Folder state at each step inside builder
AFTER: COPY pom.xml .
/app/ ├── pom.xml ← dep manifest
AFTER: COPY . . (source added)
/app/ ├── pom.xml ├── src/ │ └── main/java/... ← .java files
AFTER: mvn clean package (build done)
/app/ ├── pom.xml ├── src/ ├── target/ │ └── spring-petclinic-3.x.x.jar ← THE OUTPUT │ ├── classes/ (compiled .class files) │ └── *.original
FINAL IMAGE: Only this survives
/app/ └── app.jar ← COPY --from=builder *.jar app.jar (everything else discarded)

🏆 Interview Q&A — Multi-Stage Mastery
Q. "Why use multi-stage builds for Java?"
To separate build and runtime environments. The builder stage uses Maven (300MB+) to compile and package the JAR. The runner stage uses only a lightweight JRE (~150MB) and the JAR. Maven, source code, and build tools never appear in the final image — resulting in 75% size reduction, fewer CVEs, and faster deployments.
Q. "What goes wrong if you use CMD with a wildcard?"
CMD runs in exec form (no shell), so glob patterns like *.jar are not expanded — they're treated as literal filenames. The fix is to rename the JAR at COPY stage: COPY --from=builder /app/target/*.jar app.jar (COPY does expand wildcards), then reference the exact name in CMD.
Q. "How do you validate a Java project before Dockerizing?"
I identify the build tool (pom.xml = Maven), run mvn clean package -DskipTests locally and verify BUILD SUCCESS, check ls target/ for the exact JAR name, confirm it runs with java -jar target/*.jar, test the app in browser on the correct port, and only then write the Dockerfile. Docker doesn't fix broken apps — it packages working ones.

🔹 ARG — Build Time Variable
# ARG is only available during docker build FROM node:18-alpine ARG APP_VERSION=1.0 # ↑ default value if not passed at build RUN echo "Building version $APP_VERSION" # ↑ works at build time ✅ # Pass at build time: docker build --build-arg APP_VERSION=2.0 -t myapp:v2 .
# After build completes: docker exec -it mycontainer printenv APP_VERSION # → EMPTY ❌ — ARG disappears after build
⚠️ Critical — ARG is NOT available in containers
Once the image is built, ARG values are gone. They are NEVER available when the container runs. If you need a variable at runtime, use ENV instead.
🔹 ENV — Runtime Variable
# ENV persists into container runtime ENV PORT=5000 ENV APP_ENV=production ENV NODE_ENV=production # Verify inside running container: docker exec -it mycontainer printenv PORT=5000 APP_ENV=production NODE_ENV=production # ↑ All present ✅
# Override at run time: docker run -e PORT=8080 myapp # ↑ Overrides Dockerfile ENV at runtime
✅ ENV survives into the container
ENV values are baked into the image and available in every container run from it. They can be overridden at runtime with -e KEY=VALUE in docker run.
Combining ARG + ENV — Real Production Pattern
# Use ARG to parameterize FROM (very powerful) ARG NODE_VERSION=18 FROM node:${NODE_VERSION}-alpine # ↑ Now you can build with: --build-arg NODE_VERSION=20 ARG APP_VERSION=1.0.0 # Pass build arg into runtime ENV ENV APP_VERSION=${APP_VERSION} # ↑ Now runtime CAN see the version ENV NODE_ENV=production ENV PORT=5006

🚨 Non-Root User — MANDATORY in Production
❌ Default — container runs as root
docker exec -it mycontainer sh whoami root ← ❌ SECURITY RISK # If app is hacked → attacker gets root # Can read ANY file, escape container
✅ Production — create and use non-root user
# Alpine Linux (node:alpine / python:alpine) RUN addgroup -S appgroup && adduser -S appuser -G appgroup USER appuser # Everything after this runs as appuser
# Debian/Ubuntu based images: RUN useradd -r -s /bin/false appuser USER appuser
Why root is dangerous: If your containerized app has a vulnerability, an attacker who exploits it gets root inside the container. Root in a container can sometimes escape to the host, especially if Docker socket is mounted.
Non-root principle = blast radius reduction: Attacker gets appuser — minimal permissions. Can't read system files, can't install packages, can't modify OS.
# Verify after running container docker exec -it mycontainer sh whoami appuser ← ✅ not root
⚠️ Gotcha: If USER is set before COPY/RUN, those instructions also run as appuser. If they need root (e.g., apt-get install), do those BEFORE the USER instruction.
💬 Interview: "Non-root user reduces blast radius. If the app is compromised, attacker gets limited permissions instead of root — can't read system files or escape the container."

🏆 Interview Q&A — Production Patterns
Q. "What is the difference between ARG and ENV?"
ARG is a build-time variable — available only during docker build and disappears after the image is created. ENV is a runtime variable — baked into the image and available in every container. ARG is used for parameterizing the build itself (e.g., base image version). ENV is used for application configuration (e.g., PORT, NODE_ENV).
Q. "Why run containers as non-root?"
Running as root means if the application is compromised, the attacker gets root privileges inside the container — potentially allowing container escape or access to host resources. A non-root user reduces the blast radius: the attacker gets minimal permissions, can't install packages, can't read system files, and can't pivot to the host.
Q. "Why avoid the :latest tag?"
Because :latest causes non-deterministic builds — the upstream image can silently change at any time without any change to your code. Pinning to an exact version like node:18.20.4-alpine ensures every build uses the exact same base, giving reproducible builds and preventing unexpected CI/CD failures.
